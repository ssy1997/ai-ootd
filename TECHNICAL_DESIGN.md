# AI OOTD 小程序技术架构与详细设计文档

## 1. 核心 API 数据契约 (API Contract)

### 1.1 核心汇总接口 (`clothing.getSummary`)
为了确保前端页面（衣橱筛选、录入页动态选项、资产看板）的统一，该接口返回结构锁定为：
```json
{
  "code": 200,
  "data": {
    "stats": { "total": 0, "utilization": 0 },
    "categories": ["上装", "下装", "..."],
    "occasions": ["通勤", "约会", "..."],
    "colors": ["黑色", "白色", "..."],
    "styles": ["圆领", "修身", "..."],
    "materials": ["棉", "羊毛", "..."]
  }
}
```
*注：严禁返回旧版的 `byCategory` 等字典对象，全量采用 Array 类型以便于前端遍历渲染。*

---

## 2. 纵向功能同步准则
- **修改流程**: 修改 Cloud Action $\rightarrow$ 全局 `grep` 检索 Call Site $\rightarrow$ 同步更新 `Page.ts` $\rightarrow$ 验证全站相关入口。
- **降级保护**: 对于聚合查询，必须处理空集情况，返回 `[]` 而非 `null` 或导致 500 错误。

---

## 3. UI/UX 极致去重逻辑
- **分拣清单**: 承担了“命名、切换、删除”三重职责。
- **智能降级**: 当识别结果只有 1 件时，前端逻辑自动隐藏分拣列表，将焦点锁定在唯一项，保持极简感。

---

## 4. 数据库设计 (Schema 2.0)
- **标准化**: 所有服饰属性（除名称外）均以 String Array 形式存储，确保可以利用数据库的高级聚合查询进行秒级汇总。
- **一致性**: `lastWornDate` 是时间追踪的唯一标准字段。
